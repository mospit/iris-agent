Every screen a corporate SAP user sees — from opening the web app for the first time to uploading a recorded VBS script and watching it become a reusable workflow.

1

User opens the web app

sap-agent.yourcompany.com — browser or Teams tab

The user navigates to the company-hosted URL (or clicks the pinned Teams tab). Since this is behind corporate SSO, they see a login screen — one click with their Microsoft work account.

What happens behind the scenes

Azure AD / Entra ID authenticates the user. The backend creates a session and checks if the SAP GUI bridge is connected. No API keys are shown — the server handles LLM auth via environment variables. The user's identity determines which SAP systems and workflows they can access.

Where the .vbs file comes from

In SAP GUI, the user goes to the toolbar ⚙️ → Script Recording and Playback → starts recording → performs their transaction manually → stops recording. SAP saves a .vbs file full of session.findById("wnd[0]/usr/...").text = "value" calls. This is the raw material.

Attachment handling is free

pi-web-ui's attachment system handles the file preview, size display, remove button, and sends the file content alongside the user's text message to the agent. We just need the sap_upload_vbs tool to know what to do when it receives a .vbs attachment.

5

Agent parses the VBS — deterministic step

No LLM needed here — pure code parsing

The agent calls the sap_upload_vbs tool. The VBS parser runs immediately — no LLM call. It regex-extracts every session.findById(...) line, identifies the T-code, groups actions into logical steps, and detects which values are hard-coded literals (candidates for parameterization). The user sees a streaming status update.

User stays in control

The user can adjust anything before saving. They might say "Keep order type OR hard-coded, I only want to change the customer and material." The LLM updates the workflow definition accordingly. Nothing is saved until the user explicitly confirms.

7

User confirms — workflow is saved

Workflow JSON stored, immediately available

The user clicks "Looks good, save it." The agent saves the workflow JSON to the backend's workflow storage (Azure Blob or local filesystem). The workflow immediately appears in the Workflows sidebar and is available for execution by any authorized user.

Deterministic execution with a human gate

Steps 1–4 executed without any LLM involvement — pure code driving SAP GUI via the bridge. The executor checked an assertion after every step (title correct, no error in status bar, field exists). Step 5 is a gate — the executor paused and returned control to the user. Nothing is saved until they click "Save order."

Complete flow summary

9 steps, ~3 minutes for the first-time VBS upload, ~15 seconds for repeat runs
The full journey

Step 1 — Open app, sign in with Microsoft SSO (one click).
Step 2 — See the chat interface with welcome message.
Step 3 — Drag .vbs file onto the chat (or click 📎).
Step 4 — Type "Convert this recording into a workflow" + Send.
Step 5 — Agent parses VBS deterministically — shows step count, field count, T-code.
Step 6 — LLM suggests parameter names — user reviews the workflow summary.
Step 7 — User confirms — workflow saved, available immediately.
Step 8 — User runs the workflow with real data — sees live step progress — confirms save.
Step 9 — Done. Document number returned. Workflow reusable forever.

For repeat use (no VBS upload)

Steps 1–2 happen once per session. Steps 3–7 happen once per new workflow. For everyday use, the user just does Step 8–9: type what they want → watch execution → confirm → done. That's a 15-second interaction for a transaction that used to take 2–5 minutes of manual clicking through SAP screens.
