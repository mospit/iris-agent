SAP GUI Agent v3
Corporate Web App + Teams

Web-first via pi-web-ui. No CLI or TUI — corporate users get a chat interface they already understand. Integrates with Microsoft Teams as a bot. Supports VBS script upload from SAP's recorder. Taps into existing corporate AI (Azure OpenAI / Copilot). Reuses everything we can from pi-mono.

01 What We Reuse vs. What We Build

Maximum leverage from pi-mono. We only build SAP-specific pieces.

Corporate users get a familiar chat UI. The entire frontend is pi-web-ui's <agent-interface> component with SAP-specific tools wired in.

document.getElementById('app').appendChild(chat);

src/teams-adapter.ts — Teams bot wired to the same agent 
import { Agent } from '@mariozechner/pi-agent-core';
import { getModel } from '@mariozechner/pi-ai';
import { sapTools, sapSystemPrompt } from './sap-tools';
import { Application } from '@microsoft/teams-sdk';

const app = new Application();

// Per-user agent sessions (keyed by Teams user ID)
const sessions = new Map<string, Agent>();

function getOrCreateAgent(userId: string): Agent {
  if (!sessions.has(userId)) {
    sessions.set(userId, new Agent({
      initialState: {
        systemPrompt: sapSystemPrompt,
        model: getModel('azure-openai', 'gpt-4o'),
        messages: [],
        tools: sapTools,
      },
    }));
  }
  return sessions.get(userId)!;
}

app.message('/sap', async (ctx) => {
  const agent = getOrCreateAgent(ctx.activity.from.id);
  const userText = ctx.activity.text;

  // Handle VBS file attachment
  if (ctx.activity.attachments?.length) {
    // Download file, pass to sap_upload_vbs tool
  }

  // Run agent — same tools, same workflow executor
  agent.subscribe((event) => {
    if (event.type === 'message_end') {
      // Convert response to Adaptive Card
      ctx.sendActivity(toAdaptiveCard(event.message));
    }
    if (event.type === 'tool_call' && event.tool === 'gate') {
      // Send confirmation card with [Approve] [Reject] buttons
      ctx.sendActivity(toConfirmationCard(event.data));
    }
  });

  await agent.prompt(userText);
});

06 VBS Script Upload & Conversion

Users record a macro in SAP GUI, upload the .vbs file, and the system converts it to a reusable deterministic workflow.

1. User records in SAP GUI

User opens SAP GUI → clicks Script Recording → performs the transaction manually → stops recording. SAP generates a .vbs file with every session.findById(...) call.

2. User uploads .vbs to the web app or Teams

Drag and drop in the chat (web app uses pi-web-ui attachments, Teams uses file attachment). User says: "Convert this recording into a reusable workflow."

3. VBS Parser extracts structured steps

The sap_upload_vbs tool parses the VBScript deterministically (regex + AST, no LLM needed). It extracts: T-code, field IDs, literal values, button presses, virtual key sends. Groups them into logical steps.

4. LLM identifies parameters

The LLM reviews the extracted steps and identifies which literal values should become parameters (e.g., "1000" in the vendor field → ${vendor}). It also suggests assertion types for each step and adds a confirmation gate before saves.

5. User reviews & confirms

The agent presents the generated workflow as a structured summary: "I found 6 steps, 4 parameters (vendor, material, quantity, plant), 1 save action. Shall I save this as create_purchase_order?" User confirms, workflow is stored.

6. Workflow is now reusable

Anyone can now say "Run create_purchase_order with vendor 2000, material SCREW-M6, quantity 500, plant 2000" and the deterministic executor runs it — same steps every time.

07 Agent Tool Definitions

Four tools registered with pi-agent-core. The LLM sees these and uses them.

ToolPurposeDeterministic?When Used
sap_list_workflows Returns all available workflow IDs, descriptions, and parameter schemas ✓ Yes LLM calls this first to discover what it can do
sap_execute_workflow Runs a workflow by ID with provided parameters. Returns success/fail + extracted data. ✓ Yes — the executor is pure code Main execution path. LLM provides workflow ID + params, executor does the rest.
sap_upload_vbs Parses an uploaded VBS file, extracts steps, then asks LLM to parameterize. Saves new workflow. Hybrid — parsing is deterministic, parameterization uses LLM When user uploads a .vbs file attachment
sap_recover Constrained recovery actions: dismiss popup, fill unexpected field, retry step, or abort. ✓ Yes — actions are from a fixed set Only during error recovery when an assertion fails

Network Consideration

The SAP GUI bridge must run on a machine with SAP GUI installed (typically inside the corporate network or on a Citrix/RDP server). The backend connects to the bridge via WebSocket over the corporate network. For cloud-hosted backends, a VPN or Azure ExpressRoute is needed. Alternatively, host the entire backend on-premises alongside the bridge.

10 Build Plan — Weeks, Not Months

We're reusing 70%+ of the stack. Here's what we actually build.

Week 1-2: SAP Bridge + Workflow Executor

Python bridge with COM connection. WebSocket server. Workflow JSON schema. Executor that handles transaction, fill_fields, press, sendVKey, read_status, assertions. Test with one workflow (ME21N) on sandbox.

Week 3: Web App with pi-web-ui

Scaffold web app with pi-web-ui components. Wire up pi-agent-core Agent with SAP tools. Connect to Azure OpenAI (or whatever the company uses). Add pi-proxy for API key management. End-to-end: user chats → workflow executes → result appears.

Week 4: VBS Upload + Converter

Build VBS parser (regex-based, deterministic). Wire as sap_upload_vbs tool. LLM parameterization step. Test with 5+ recorded scripts from different transactions. Store generated workflows.

Week 5: Teams Bot

Register bot with Azure Bot Service. Build Teams adapter using Teams SDK. Adaptive Cards for responses + confirmation gates. File attachment handling for VBS uploads. Deploy to same App Service as web app.

Week 6: Batch + Polish

Batch processing from XLSX/CSV uploads. Confirmation gates in both web and Teams. Error recovery flow. Workflow library sidebar in web app. Corporate branding pass. SSO integration.

Week 7-8: Testing + Pilot

Test against real SAP system with 10+ workflows. Expand VBS converter for edge cases (table controls, popups, tree nodes). Pilot with 5-10 users. Collect workflow recordings from users to build the library. Harden error handling.
